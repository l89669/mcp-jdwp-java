## Where we actually use it

- **Track complex bugs to their source** — follow a wrong value back to the line that wrote it, across threads and frames
- **Prove a hypothesis with no test for it** — the agent reads the code, forms a theory, then checks it against the live JVM instead of guessing
- **Diagnose a live call end-to-end** — run the app, let the agent fire a REST / GraphQL request and inspect what the server does with it: real-dataset bugs, performance glitches
- **Debug straight on staging** — attach over JDWP to a running K8s pod; logpoints — and even thread-suspending breakpoints — don't trip the infra into restarting the container

<small class="muted">In daily use on our own projects for about six weeks — no guarantees, but it works for us. <em>(MIT-licensed.)</em></small>

<small class="muted">Full disclosure: this plugin — and these slides — were <strong>vibe-coded</strong> by the agent itself, with me as overseer.</small>

Note:
- Not a recap — these are the four things we genuinely reach for it for, day to day
- Bug-to-source: the watchpoint / eval workflow from the previous slides — follow a wrong value back through frames and threads to the line that wrote it
- Hypothesis without a test: the agent has read the code and has a theory, but there's no failing test to confirm it — attach to the live JVM and check the theory directly rather than writing a throwaway test
- Live call: run the app, agent fires a REST / GraphQL call and inspects server-side behaviour on a real dataset — catches data-shape bugs and performance glitches a unit test never sees
- Staging: JDWP attach to a running pod — logpoints never suspend, and in our setup even thread-stopping breakpoints don't fail the liveness probe / recycle the container, so you debug the reported issue in place with no redeploy
- Honest-status line is deliberate: ~six weeks of self-use, no guarantees — sets expectations, earns trust
